Spring 依赖注入顺序详解

Spring 框架支持多种依赖注入方式，它们的执行顺序和优先级是开发者需要清楚理解的重要概念。下面我将详细解析各种注入方式的执行顺序及其相互关系。

一、注入方式分类

Spring 主要支持以下几种依赖注入方式：

  1. 构造器注入 - 通过构造函数
  2. 字段注入 - 直接注入字段
  3. Setter 注入 - 通过 setter 方法
  4. 方法注入 - 任意方法的参数注入

二、注入执行顺序

1. 总体执行顺序

2. 详细顺序说明

(1) 构造器注入（最先执行）

  • 触发时机：Bean 实例化时
  • 特点：
    • 在对象创建时立即完成依赖注入
    • 强依赖必须使用构造器注入
    • 解决循环依赖的能力有限

构造器注入

  • 通过构造函数参数声明依赖。
  • @Autowired 可省略（Spring 4.3+ 单构造器时自动注入）。
  • 推荐：用 final 修饰字段，确保依赖不可变（线程安全）。

(2) 字段注入（其次执行）

  • 触发时机：Bean 实例化后，初始化前
  • 特点：
    • 通过反射直接设置字段值
    • 不需要 setter 方法
    • 测试时不太方便

(3) Setter 注入（然后执行）

  • 触发时机：字段注入之后，初始化方法之前
  • 特点：
    • 灵活性高，可以重新注入
    • 适合可选依赖

(4) 任意方法注入（最后执行）

  • 触发时机：初始化阶段
  • 特点：
    • 可以标注在任何方法上
    • 方法名无限制

四、特殊场景处理

1. 循环依赖下的注入顺序

解析：

  1. Spring 会先尝试构造器注入 ClassB，发现需要 ClassA
  2. 转而先创建 ClassA，完成其构造
  3. 对 ClassA 进行 setter 注入 ClassB
  4. 最后完成 ClassB 的构造器注入

2. @Autowired 和 @Resource 的优先级

  • @Autowired：按类型匹配，是 Spring 原生注解
  • @Resource：先按名称再按类型，是 JSR-250 标准

当两者同时存在时，默认情况下：

  1. @Autowired 构造器注入最先执行
  2. @Resource 字段/setter 注入随后执行
  3. @Autowired 字段/setter 注入最后执行

五、最佳实践建议

  1. 强制依赖使用构造器注入

    • 保证依赖不为 null
    • 方便单元测试
    • 推荐使用 Lombok @RequiredArgsConstructor

  2. 可选依赖使用 setter 注入

    • 允许重新配置
    • 适合有默认实现的依赖

  3. 避免混合使用多种注入方式

    • 同一 Bean 中尽量统一使用一种注入方式
    • 构造器注入 + setter 注入混用会增加复杂度

  4. 测试友好的设计

  5. 循环依赖解决方案

    • 优先考虑重构代码消除循环依赖
    • 必须使用时，使用 setter 注入替代构造器注入
    • 使用 @Lazy 延迟加载

理解 Spring 的依赖注入顺序对于解决复杂的依赖问题和优化应用架构非常重要。合理选择注入方式可以使代码更加健壮和可维护。
