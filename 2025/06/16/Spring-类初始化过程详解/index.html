Spring 类初始化过程详解

Spring 框架中类的初始化是一个复杂而精细的过程，涉及多个阶段和扩展点。下面我将详细解析 Spring 管理的类从创建到完全可用的完整生命周期。

一、核心初始化阶段

1. 实例化阶段

触发时机：当容器需要创建 Bean 实例时

过程：

  1. 通过反射调用构造函数创建对象实例
  2. 如果是工厂方法创建的 Bean，则调用指定的工厂方法
  3. 处理构造器注入（如果有）

相关扩展点：

  • InstantiationAwareBeanPostProcessor.postProcessBeforeInstantiation()
    • 在实例化前执行，可以返回代理对象替代实际实例

2. 属性填充阶段

触发时机：实例创建后，初始化前

过程：

  1. 处理 @Autowired、@Value、@Resource 等注解的字段注入
  2. 处理 XML 中配置的 <property> 元素
  3. 应用 BeanPostProcessor 的预处理

相关扩展点：

  • InstantiationAwareBeanPostProcessor.postProcessAfterInstantiation()
    • 在属性设置前执行，可以阻止默认属性填充
  • InstantiationAwareBeanPostProcessor.postProcessProperties()
    • 对属性值进行后处理

3. 初始化阶段

触发时机：属性注入完成后

过程：

  1. 调用 Aware 接口方法（BeanNameAware, BeanFactoryAware 等）
  2. 执行 BeanPostProcessor.postProcessBeforeInitialization()
  3. 调用初始化方法：
    • @PostConstruct 注解方法
    • InitializingBean.afterPropertiesSet()
    • 自定义 init-method
  4. 执行 BeanPostProcessor.postProcessAfterInitialization()

三、关键扩展点详解

1. Aware 接口系列

Spring 提供了一系列 Aware 接口，用于在初始化阶段获取容器基础设施：

接口 作用 注入时机
BeanNameAware 获取 Bean 名称 初始化早期
BeanFactoryAware 获取 BeanFactory 初始化早期
ApplicationContextAware 获取 ApplicationContext 初始化早期
EnvironmentAware 获取 Environment 对象 初始化早期
ResourceLoaderAware 获取资源加载器 初始化早期

2. BeanPostProcessor 执行顺序

  1. postProcessBeforeInstantiation (如果存在)
  2. 实例化对象
  3. postProcessAfterInstantiation (如果存在)
  4. 属性填充
  5. postProcessBeforeInitialization (按优先级顺序)
  6. 初始化方法 (@PostConstruct, afterPropertiesSet, init-method)
  7. postProcessAfterInitialization (按优先级顺序)

3. 初始化方法执行顺序

  1. @PostConstruct 注解方法
  2. InitializingBean.afterPropertiesSet()
  3. 自定义的 init-method (XML 或 @Bean(initMethod="..."))

四、特殊场景处理

1. 循环依赖的初始化

Spring 通过三级缓存解决 setter 注入的循环依赖：

  1. 一级缓存：存放完全初始化好的 Bean
  2. 二级缓存：存放早期暴露的原始 Bean（已实例化但未初始化）
  3. 三级缓存：存放 Bean 工厂，用于生成早期引用

2. 代理对象的初始化

对于需要 AOP 代理的 Bean，初始化过程有特殊处理：

  1. 创建原始 Bean 实例
  2. 执行 @PostConstruct 等初始化方法（在原始对象上）
  3. 创建代理对象包装原始对象
  4. 后续调用都通过代理对象

六、常见问题排查

1. 初始化方法未执行

可能原因：

  • Bean 没有被 Spring 管理（缺少 @Component 等注解）
  • 配置了 @Lazy 但从未实际使用该 Bean
  • 初始化方法被错误覆盖

2. 循环依赖导致初始化失败

解决方案：

  • 改为 setter 注入而非构造器注入
  • 使用 @Lazy 延迟加载其中一个依赖
  • 重构代码消除循环依赖

3. 初始化顺序不符合预期

解决方案：

  • 使用 @DependsOn 明确声明依赖关系
  • 实现 PriorityOrdered 或 Ordered 接口调整 BeanPostProcessor 顺序
  • 拆分配置类控制加载顺序

Spring 的类初始化过程提供了丰富的扩展点和精确的生命周期控制，理解这些机制对于开发复杂的 Spring 应用至关重要。合理利用这些特性可以实现更灵活、更可靠的组件设计。
