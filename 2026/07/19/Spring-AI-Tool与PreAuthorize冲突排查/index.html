Spring AI：@Tool 与 @PreAuthorize 同方法并存导致工具不生效

运行环境

本文基于以下本地环境排查与验证（05tools 模块）：

组件 版本
JDK Oracle JDK 17.0.19（LTS，java version "17.0.19"）
Maven Apache Maven 3.9.11
Spring Boot 3.2.5
Spring Framework 6.1.6（由 Spring Boot 3.2.5 依赖管理引入）
Spring AI 1.0.0
Spring AI Alibaba 1.0.0.2（spring-ai-alibaba-starter-dashscope）
大模型 阿里云百炼 / DashScope，默认 Chat 模型 **qwen-plus**（05tools 未显式配置 spring.ai.dashscope.chat.options.model，使用 starter 默认值 DEFAULT_DEPLOYMENT_NAME）

OS 参考：macOS 15.7.7（aarch64）。父工程 pom.xml 中 java.version=17。

现象

在 Spring AI 项目中，通过 ChatClient.Builder.defaultTools(toolService) 注册工具类，方法上已标注 @Tool / @ToolParam，但实际运行时：

  • 大模型侧看不到任何可用工具（请求里没有 tool definitions）
  • 涉及退票、查天气等本应触发 Function Calling 的对话，模型只会「空口回答」
  • 日志里也看不到工具被调用

代码写法看起来「完全正确」：

同类写法在没有引入 Spring Security 方法级鉴权的模块里是正常的，一加上 @PreAuthorize 就全挂。

原因分析

直接原因

@Tool 与 @PreAuthorize 写在了同一个 Bean 的方法上。开启 @EnableMethodSecurity 后，Spring Security 会为该 Bean 创建 CGLIB / JDK 动态代理。

Spring AI 的 defaultTools(Object...) 底层通过 MethodToolCallbackProvider 对传入对象做反射扫描，查找带 @Tool 的方法。扫描落在代理类上时：

  • getDeclaredMethods() 拿到的是代理子类方法
  • 原目标类上的 @Tool 不会出现在代理方法上
  • 结果：一个 @Tool 都扫不到 → 工具列表为空 → 「@Tool 没生效」

这不是大模型「不听话」，而是 工具压根没注册进 ChatClient。

调用链（出问题前）

为什么「类上只有一个方法加了 PreAuthorize，所有 Tool 都失效」

方法级安全是 按 Bean 做代理 的。只要类里存在任意 @PreAuthorize / @Secured 等方法安全注解，整个 ToolService Bean 都会被代理。

因此即使 getAirQuality() 只有 @Tool、没有权限注解，只要同类里的 cancel() 带了 @PreAuthorize，两个工具都会扫不到。

与官方问题的对应关系

这与 Spring AI 社区反馈一致：Unable to use @Tool with @PreAuthorize #2356。框架后续版本对 AOP 代理有过增强，但实践中最稳妥的做法仍是：工具声明与权限校验分层，不要绑在同一组方法注解上。

处理方案

推荐方案：职责拆分（本次采用）

职责 放在哪 注解
暴露给大模型的工具入口 ToolService 只保留 @Tool / @ToolParam
真实业务 + 权限 TicketService（或独立 Secured 服务） @PreAuthorize 等

ToolService 不要再被方法安全代理；它只做「工具门面」，内部委托给已鉴权的业务 Bean。权限仍通过 Spring 代理生效（调用的是注入的 TicketService 代理，而不是 this）。

备选方案（按场景选用）

  1. 编程式注册 MethodToolCallback
    自己用 ReflectionUtils.findMethod(ToolService.class, ...) 拿目标类方法，再 .toolObject(代理Bean)。
    注解扫描绕开了，但要手写 ToolDefinition（name / description / inputSchema），@Tool / @ToolParam 不会自动生效。

  2. 升级 Spring AI 并确认代理解包
    新版本对 AOP 有改善，仍建议不要依赖「同方法叠两个注解」作为长期设计。

  3. 方法内手动鉴权
    在 @Tool 方法里用 SecurityContextHolder / AuthorizationManager 判断。能跑，但不如声明式清晰，且难做「按角色动态裁剪工具列表」。

如何避免类似问题

设计原则

  1. 工具门面与安全边界分离

    • @Tool 类：薄封装，无方法级安全 / 事务 / 缓存等会强制代理的注解（或确保不会因此变成「扫不到注解的代理」）。
    • 业务类：放 @PreAuthorize、@Transactional 等。

  2. 不要假设「代理对注解透明」
    凡是靠 反射读注解（Spring AI @Tool、部分自定义扫描器）的能力，都要警惕 CGLIB / JDK 代理。同类问题还包括：对代理 Bean 直接 getDeclaredMethods()、在自调用 this.xxx() 时期望 AOP 生效等。

  3. 注册方式与注解语义对齐

    • 声明式：defaultTools(pojo) → 依赖类上真实存在的 @Tool。
    • 动态式：defaultToolCallbacks(...) / 手写 MethodToolCallback → 必须自己补全 ToolDefinition，别指望再读一遍 @Tool。

  4. 联调时先确认「工具有没有进请求」
    排查顺序建议：
    1）启动后打断点或打日志，看 ToolCallback 列表是否为空；
    2）再看 LLM 请求 JSON 是否带 tools；
    3）最后才查 Prompt、模型能力、权限异常。
    避免一上来怀疑「模型不调用工具」。

  5. 鉴权与 HTTP 放行策略一致
    工具注册修好后，若接口 permitAll 且未登录，带 @PreAuthorize("hasRole('ADMIN')") 的业务仍会 AccessDenied。这是预期行为，需用登录用户（如 admin）验证退票路径，用天气等无权限工具验证注册是否成功。

快速自检清单

  • 带 @Tool 的类上，是否还有 @PreAuthorize / @Secured / @RolesAllowed？
  • 该类是否因 @Transactional、自定义 AOP 等被代理？调试时看 toolService.getClass().getName() 是否含 $$SpringCGLIB$$ / $Proxy。
  • 注入的是 Spring Bean，而不是 new ToolService()（后者无注入，也无代理，但依赖会空指针）。
  • 动态组装工具时，是否误以为 @Tool 仍会自动生成 schema？

小结

@Tool「不生效」，在本案例中并不是注解写错，而是 @PreAuthorize 触发的 AOP 代理挡住了 Spring AI 对 @Tool 的反射发现。把权限下沉到业务服务、让工具类保持「可被扫描的朴素 Bean」，即可同时保留 Function Calling 与方法级鉴权。

后续凡是「框架靠反射读注解 + Spring 又可能代理该 Bean」的组合，都按同一思路：注解消费方看到的类，必须是带注解的那一层；需要代理的能力，放到下一层委托对象上。
