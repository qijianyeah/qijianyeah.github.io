Kafka 集群资源评估

本文记录一次 Kafka 集群的容量评估过程，覆盖 CPU / 内存 / 磁盘 三个维度。
核心结论：单节点 16C 32G + 2T 磁盘，集群 5 节点、副本 3、保留 7 天。
Kafka 的瓶颈在磁盘容量，CPU 主要消耗在副本复制（若开启压缩还需额外考虑压缩开销）。

假设 取值 说明
峰谷比 5 倍 按平均速率的 5 倍估算峰值
热数据缓存窗口 15 分钟（900s） Page Cache 需缓存的热数据时间窗
磁盘安全水位 65% 磁盘不能跑满，按使用率不超过 65% 预留

说明：Kafka 的主要资源消耗在副本复制；若开启压缩，还需额外考虑压缩 / 解压的 CPU 开销。

二、CPU 评估

Kafka 主要的 CPU 消耗来自副本复制（leader 向 follower 同步数据），若开启压缩还会叠加压缩 / 解压开销。结合复制与压缩的开销预留，单节点 CPU 取：

CPU = 16 核

六、不同交易量换算表

以日交易量 1000W（日数据量 0.3 TB）为基准，磁盘需求随交易量线性增长（副本 3、保留 7 天、水位 65%）：

日交易量 日数据量 集群总磁盘（×3 副本 ×7 天 ÷0.65） 单节点磁盘（5 节点）
1000W 0.3 TB ≈ 9.7 TB ≈ 2 T
1500W 0.45 TB ≈ 14.5 TB ≈ 3 T
2500W 0.75 TB ≈ 24.2 TB ≈ 5 T

若开启 Kafka 压缩（lz4 约 0.4 压缩比），磁盘需求可进一步下降，本表未计入。

七、评估结论

维度 结论 说明
CPU 16 核 主要消耗在副本复制（+压缩）
内存 32 GB 16G Page Cache 热数据 + 8G JVM 堆，取 32G
磁盘 2 T / 节点 核心瓶颈维度
节点 / 副本 5 节点 / RF=3 容忍 2 副本故障
保留 7 天 —

一句话总结：按日交易量 1000W（日数据量 0.3TB）、副本 3、保留 7 天、磁盘水位 65% 估算，单节点 16C 32G + 2T、集群 5 节点即可满足；瓶颈在磁盘容量而非 CPU / 内存，且本次未计入压缩，磁盘侧仍有余量。
